2024-09 – 2026-04

補助から限定的な代行へ

sec-3 のボトルネックは、信頼性と論理思考能力でした。
ChatGPT で入口は壊れたけれど、長い仕事を最後までやりきれるか、論理的な一貫性があるか、という点ではまだ弱かった。

2024年の秋、その壁に正面からぶつかるような変化が起きます。
モデルは「答える前に、もっと考える」ようになり、改善のサイクルそのものが猛烈に速くなり、ついにはまとまった仕事を自律的にこなし始めました。

4-1. 加速

2024年9月、OpenAI は o1-preview を発表しました。s4-01

o1 系モデルの特徴は、問題に答える前に内部でより長く推論することです。OpenAI 自身はこれを reasoning model と呼びました。数学やコーディング、科学分野で従来モデルより明らかに強い結果を出しています。s4-01

ここでいう推論とは、「途中の考えを積み上げながら答えを作る」こと。sec-2 で見た "Let's think step by step" を覚えていますか。あの手法は、ユーザーが賢いプロンプトを書くことでモデルの隠れた力を引き出していました。o1 系では、そういう手法をモデルの性質そのものに埋め込んだと考えるのが近いでしょう。

OpenAI は内部の実装詳細を全部は明かしていません。でも system card や公開研究からは、強化学習と長めの熟考を組み合わせた方向性がうかがえます。s4-02 2025年初頭には s1 や DeepSeek-R1 のような公開研究が現れ、「テスト時により長く考えさせると推論能力が伸びる」という流れを外部から再現・一般化しようとしました。s4-03 s4-04

この発想は、test-time scaling（テスト時スケーリング） と呼ばれるようになります。

sec-2 で見たスケーリングは、訓練時にどれだけ計算資源、データ、パラメータを投入するか、でした。test-time scaling は、それに加えて、推論時にどれだけ計算を使わせるか を新しい調整つまみにします。たとえるなら、受験勉強の量が従来のスケーリングだとすれば、試験本番で「どれだけ考える時間を与えるか」が新たに加わったようなものです。勉強量が同じでも、本番でじっくり考えれば正答率が上がる問題はありますよね。

OpenAI は 2024年12月に o1 の正式版を公開し、そのわずか15日後には o3 を発表しました。s4-05 s4-06
この「もう次が来たのか」という感覚――これがこの時期を象徴しています。

なぜそんなに速いのか。
答えのもう半分は、性能向上の主戦場が 事前学習だけでなく、事後学習にもはっきり移った ことにあります。

事前学習とは、膨大なデータで巨大モデルの土台を一から作る工程です。数か月単位の大プロジェクトになります。対して事後学習は、その土台モデルの上に、指示追従、推論、ツール利用、安全性といった性質を追加で鍛える工程です。事前学習ほどの時間をかけずに、大きな体感向上を得られることがある。

この違いは、製品のリリース間隔を変えます。

2023年ごろは「次の大きな基盤モデル」が来るまでしばらく待つ感じがありました。ところが 2025年になると、半年どころか数か月前の最先端モデルがもう一世代前に見える。実際、この18か月で OpenAI は少なくとも8世代、Anthropic は少なくとも10世代のモデルを送り出しています。名前の付け方やマーケティングを差し引いても、改善の頻度が目に見えて上がっていることは確かです。

改善サイクルが短くなると、社会の側が追いつく前に、また次のモデルが来ます。

コラム: test-time scaling って何？

人間に難しい問題を出すとき、すぐ答えさせるのと、紙と時間を渡して考えさせるのでは結果が変わります。
test-time scaling は、ざっくり言えばこの違いを AI に持ち込む発想です。

従来の言語モデルは、質問を受けると比較的まっすぐ答えを吐きます。
もちろん内部では計算していますが、発想としては「一気に次のトークン列を出す」に近い。これに対して reasoning model では、内部でいったんより長い探索や比較、途中方針の見直しを行い、そのうえで最終回答を作ろうとします。

なぜこれで強くなるのでしょうか。
単純な質問なら即答で十分です。でも数学の証明、複雑なデバッグ、長い仕様書の読解のような問題では、「最初に思いついた答え」が雑すぎることが多い。途中で仮説をいくつか試し、矛盾を潰し、別解を比べたほうがよい。こういう問題では、推論時に使う計算量を増やすこと自体が性能向上に効きます。

これは、sec-2 の Chain-of-Thought とどう違うのでしょうか。
Chain-of-Thought は、途中の考え方を出力として書かせるテクニックでした。test-time scaling は、途中でより多くの計算や探索を行わせるという、もう少し広い概念です。途中思考がそのまま見える場合もあれば、見えない場合もあります。

ここで注意したいのは、長く考えたからといって必ず賢いわけではないことです。
人間でも、変な仮定にハマって長考すると、むしろ間違いを深掘りすることがありますよね。AI でも同じです。だから test-time scaling は「第二の魔法の杖」ではなく、「うまく使えば効く追加のレバー」と見るのが近いでしょう。

それでも重要なのは、スケーリングの対象が訓練時だけではなくなったことです。
これにより、フロンティアモデルの改善サイクルは、巨大な事前学習を待たずとも、事後学習と推論設計で回せる部分が増えました。sec-4 で加速感が強まるのは、このためです。

4-2. エージェント: 不完全な同僚がやってきた

この時期にいちばん目に見えて変わったのは、巨大言語モデルの使われ方です。

sec-3 までのモデルは、基本的に「一回の質問に一回答える」存在でした。もちろん sec-3 で見た function calling や RAG はありましたが、それでも構造はかなり単発的でした。ここで前に出てくるのが、エージェントという言葉です。

エージェントとは何か。
この文脈では、「目標を与えられると、複数の手順をまたいで、必要な道具を選び、途中で状態を確認しながら、ある程度自律的に仕事を進める AI システム」くらいの意味です。会話の返事を一回返すだけではなく、ファイルを読み、コマンドを打ち、ブラウザを開き、テストを走らせ、失敗したらやり直す。そうしたループを回すのがエージェントです。

この波が最初に大きく立ったのは、コーディングでした。

理由はかなりはっきりしています。
第一に、ソフトウェア開発には検証ループがあります。コードを書いたら、テストを回せる。エラーが出たら直せる。つまり「間違ってもあとで確かめられる」。
第二に、操作対象がほぼ全部デジタルです。ファイル、ターミナル、Git、ブラウザ、エディタ。AI が触る対象をプログラム経由で用意しやすい。
第三に、価値がわかりやすい。バグ修正、コード生成、リファクタリング。どれも時間短縮効果を体感しやすい。

同じ理由が文章や企画書にあてはまるかというと、そこまで簡単ではありません。「どこがダメか」を機械的に検出しにくいからです。だからコーディングが最初の主戦場になったのは、偶然ではありませんでした。

初期には、AI を深く統合したコードエディタや、AI がファイル編集やコマンド実行を自律的に行うコミュニティ製ツールが注目を集めました。s4-22 s4-23

ここで少し用語を整理しておきましょう。
IDE は Integrated Development Environment、つまりプログラマー向けの高機能メモ帳のようなものです。コードを書き、補完し、テストを動かし、Git を触る作業が一か所にまとまっています。
CLI は Command Line Interface。画面上のボタンの代わりに、文字で命令を打ち込みます。エンジニア以外にはやや縁遠いですが、命令の形がはっきりしていて、自動化や AI との相性がよい。

面白いのは、エージェントの「活動場所」が短期間で移り変わったことです。
最初は IDE 付属のチャット欄。次に CLI。各社が AI がターミナル作業を直接支援する形式を次々と投入しました。s4-10 s4-24 s4-25
さらに進むと、差分確認や成果物レビューを中心にした、よりタスク志向のアプリが現れます。s4-25 s4-26
これは、モデルの能力だけでなく、どんな UI がもっとも「任せやすい」か を各社が急いで探っていたことを示しています。sec-3 で ChatGPT の UI が普及の鍵だった話を思い出すと、ここでもまた UI が勝負を分けているのが見えますね。

ただし、エージェントが登場したからといって、いきなり人間の仕事を丸ごと置き換えたわけではありません。
むしろ現実は、「驚くほど役に立つ局面もあるが、まだかなり壊れやすい」です。

それをうまく捉えているのが、METR（Model Evaluation and Threat Research）の Time Horizon という見方です。これは、AI がどれだけ長いタスクを成功裏に完遂できるかを、人間がかかる時間に換算して測る指標です。2025年3月の報告では、50% 成功率で見た場合にこの指標が過去6年で約7か月ごとに倍増してきたとされます。Claude 3.7 Sonnet は50%成功率で約1時間相当のタスクまで到達した一方、信頼して任せられる長さで見ると、まだ「数分程度まで」が中心だとも述べられています。s4-27

この二つの数字は、いかにも "ジグザグ" です。
能力の伸びは明らかに速い。けれど、現場で「安心して丸投げできる」にはまだ足りない。

sec-1 で AI Dungeon が「すごいけど、すぐ壊れる」と評されたことを覚えているでしょうか。あのときは、局所的な文章生成のそれっぽさと、長期的な破綻の間にギャップがありました。ここでも構造は似ています。ただし、「壊れるまでの持続時間」が桁違いに伸びたのです。

さらに、エージェントはコンテキスト長の壁にもぶつかります。
長い作業を続けるほど、履歴は伸び、読まなければいけない情報は増えます。するとモデルのコンテキスト枠を圧迫し、以前の判断や状態を保つのが難しくなる。Claude Code や Codex 系ツールでは、履歴を要約し直して圧縮する compaction のような対策が導入されました。場当たり的に見えるかもしれませんが、逆に言えば、モデルの自律行動が、周辺インフラの限界に追いつき始めた 証拠でもあります。

だからエージェントは、完璧だから広まったのではありません。
不完全でも、すでに経済価値があるから広まった。sec-2 で GitHub Copilot が「コードの次の一行を提案する」だけで価値を出したように、全部を任せられなくても、一部を速くするだけで十分に助かる場面が多かったのです。

巨大言語モデルは、この時期に初めて「一問一答の賢い会話相手」から、「まとまった作業を受け持つ不完全な同僚」へ変わり始めました。

コラム: なぜ「コーディング」が最初のエージェント市場になったのか

コーディングが「AI が最初に完全代替した仕事」…ではありません。
むしろ、「AI が不完全でも最初に大きな価値を生みやすかった仕事」でした。

一番大きいのは、正誤確認がしやすいことです。
コードを書いたらテストできます。ビルドできます。エラーが出たら直せます。つまり AI が一度ミスしても、人間や自動テストがすぐ検出しやすい。文章や企画書のように「どこがダメか」を判断しにくい仕事より、エージェントに任せやすいのです。

次に、作業対象が全部デジタルだということ。
コード、ログ、ファイル、ブラウザ、ターミナル。全部がコンピュータ上にあり、API やコマンドで触れます。AI が道具を使うための環境を用意しやすい。

さらに、一部分だけでも価値が大きいこともあります。
全部を任せられなくても、「このエラーを調べる」「この関数をリファクタする」「ドキュメントを書き直す」だけで十分助かる。つまり、部分自動化の経済性が高いのです。

sec-4 の本質は、このズレにあります。完全自律化より前に、経済的実用化が先に来たのです。

4-3. AI が研究そのものに入り込み始めた

ここまで見てきたのは、「人間が AI を使う」話でした。
AI が文章を書き、コードを補完し、ツールを呼び、ある程度長いタスクをこなす。それだけでも大きな変化です。

でも 2025年後半から 2026年にかけて、もう一歩先の事例が見え始めます。
AI が、科学研究や高度な開発そのものに、少なくとも部分的に貢献し始めたのです。

たとえば Google DeepMind は、2025年に LLM のコーディング能力を使ったアルゴリズム探索エージェントを発表し、データセンター効率の改善や数学の未解決問題への貢献を報告しました。s4-37 生物学の実験プロトコル最適化で 79倍の効率向上や、理論物理学での新しい結果への寄与も報告されています。s4-38 s4-39 サイバーセキュリティでも、脆弱性発見能力がきわめて高いモデルが現れ、防御目的での限定配備が行われています。s4-43 s4-44

もちろん、AI が単独で科学を全部やっているわけではありません。人間研究者が課題を設定し、検証し、実験を回し、論文化しています。それでも意味は大きいのです。なぜなら、「AI が論文の下書きを手伝う」を超えて、「AI が予想や改善案を出し、それが人間の検証を通って残る」段階が見え始めたからです。

そして最も象徴的だったのが、OpenAI が 2026年初頭に発表したモデルの紹介文でした。
OpenAI はこのモデルを、「自身の開発に重要な役割を果たした初のモデル」と説明しました。初期版が自分自身の訓練デバッグ、デプロイ管理、テスト結果の診断に使われたと述べています。s4-20

この一文は、大げさな未来宣言とは違います。AI が AI 開発を支援することが、未来予想図ではなく、現在形として語られ始めているのです。

ただし、バランス感覚も必要です。
完全自律の研究機関が今すぐ生まれたわけではありません。最前線のモデル開発企業自身が「非常に強いが、まだ自動化された AI R&D の閾値を超えたとは見ていない」という慎重な評価を出しています。s4-44

それでも、方向性はかなりはっきりしています。
AI は、補助ツールとして文章やコードを手伝うだけでなく、研究・開発・発見のループそのものへ入り込み始めた。まだ断片的で、検証も人間依存で、危うさもある。けれど「その可能性がある」という話ではなく、「すでに部分的に起きている」という話に変わったのです。

4-4. 評価の危機

能力がこれだけ速く伸びると、困るのは比較の仕方です。

sec-1 で見た GLUE は、言語理解の総合模試でした。sec-3 では MMLU や GSM8K が共通言語になっていました。でも sec-4 に入ると、それらが軒並み天井にぶつかります。

学校の模試で上位層がみんな98点、99点を取るようなもので、「誰が本当に強いか」が見えにくいし、改善を測る物差しとしても鈍い。

そこで研究者たちは、もっと難しい試験を作り始めます。大学院レベルの理系難問、数学オリンピック予選級の問題、実際のバグ修正をさせる実タスク評価。ところがこれらも、フロンティアモデルの前では案外早く飽和し始めました。さらには世界中の専門家が集めた超難問セットや、未解決問題まで視野に入れた評価まで登場します。s4-28 s4-29 s4-30 s4-31 s4-33 s4-34 評価の対象そのものが、教科書的な問題から現実の仕事や未公開課題へ移っているのです。

なぜそんなことが必要なのか。
最大の理由は汚染です。公開ベンチマークは、時間がたつほど訓練データに混ざりやすくなります。モデルが本当に初見で解いているのか、似た問題を見てしまっているのか、わかりにくくなる。だから新しいベンチマークは隠し問題や非公開セットを設けたり、インタラクティブな形式へ移行したりと、「漏れにくい」仕組みを探っています。

もっと根本的な話もあります。
実際の遠隔業務プロジェクトをまるごとやらせたり、44職種にわたる知識労働の成果物を作らせて人間の専門家と比べる評価も現れました。s4-35 s4-36
評価は、「模試の点数を競う」段階から、「何をどう測れば実力に近いのかを探す」段階へ入ったのです。

sec-1 では「何をもってすごいと呼んでいたか」を追いました。
sec-4 で起きているのは、その問い自体が壊れ始めている、ということです。

コラム: 「ベンチマークが壊れる」とはどういうことか

AI のベンチマークは、テストです。
テストが壊れるとは、「問題が簡単すぎる」「問題が漏れている」「点数が実力を表さなくなる」のどれか、あるいは全部です。

まずわかりやすいのが飽和。
みんなが高得点を取るようになると、差が見えません。MMLU や GSM8K で起きたのは、まさにこれです。

次に汚染。
ベンチマークが公開され、有名になり、訓練データにも評価レポートにも何度も登場すると、"モデルがそのベンチマークを既に知っている"可能性が上がります。これでは、本当に「初見で解いた」のかがあやしくなります。

さらにゲーム化も起きます。
研究者や企業が、「このベンチマークでよく見えるように」最適化し始めるのです。すると、点数は上がっても、実力が同じだけ上がったとは限りません。

最後に、ベンチマーク設計そのものの欠点があります。
GPQA Diamond には noise ceiling（問題文や選択肢設計の限界による理論上の最高点の制約）が指摘されています。s4-30
SWE-Bench Pro も、実行環境やエージェント設計で結果がかなり変わりえます。s4-31
Remote Labor Index は現実の仕事に近いのが魅力ですが、現時点では最先端エージェントの性能が「床に近い」と公式に表現されるほど低い。s4-36

だから最近の評価は、単に「もっと難しい問題集を作ろう」では終わりません。
隠し問題、private set、interactive benchmark、実際の仕事に近い成果物評価へ進んでいます。ベンチマークは依然として必要です。でも sec-4 の時代には、それを一つの点数で世界を語る道具として扱うのは、だいぶ危うくなっているのです。

  • AIME: 数学オリンピック予選級の問題。数学的推論力の指標として広く使われる。
  • SWE-Bench / SWE-Bench Pro: GitHub issue をもとに実際のバグ修正をさせる実タスク評価。s4-31
  • ARC-AGI: 人間には比較的やさしいが AI には難しいパターン推理課題。s4-32
  • Humanity's Last Exam (HLE): 世界中の専門家が集めた超難問セット。2026年4月時点で上位モデルもまだ半分に届かない。s4-28 s4-29
  • FrontierMath: とくに難しい数学問題でモデルを測る試み。Open Problems 版では未解決問題への寄与まで視野に入る。s4-33 s4-34
  • GDPval: 44職種にわたる知識労働の成果物（スライド、表計算、文書等）を作らせて評価。s4-35
  • Remote Labor Index: end-to-end の遠隔業務プロジェクトを扱う。フロンティアモデルでもまだ性能は低い。s4-36

4-5. この時期の経済的風景: インフラと労働のあいだ

能力の進歩は、お金の流れにも現れます。

まず目立つのが、データセンター投資の巨大化です。
2025年1月、OpenAI は SoftBank、Oracle などとともに Stargate Project を発表しました。今後4年間で最大5000億ドルを投じて米国内の AI インフラを整備する構想です。s4-45 xAI の Colossus は 200,000 GPU 規模の巨大計算基盤として宣伝され、Amazon と Anthropic の Project Rainier も数十万個規模のチップを用いる超大型クラスタとして進行中です。s4-46 s4-47

金額の大きさ自体よりも重要なのは、AI の競争がもはや純粋なソフトウェアの話ではなくなっていることです。
土地、電力、冷却、半導体、建設、クラウド、資本。これら全部が競争力の一部になっている。sec-3 で NVIDIA を「ゴールドラッシュのツルハシ売り」と紹介しましたが、いまやツルハシどころか鉱山そのものの巨大開発競争が始まっていまいました。

OpenAI 自身の企業規模もそれに伴って膨張しています。2025年3月に400億ドル、2026年4月にはさらに1220億ドルの資金調達を発表し、ChatGPT の週次利用者は5億人に達したと説明しました。s4-49 s4-50
もはや大型スタートアップの話ではなく、国家級インフラと結びついた産業です。

一方、既存の仕事の中への浸透 も着実に進みました。

Microsoft 365 Copilot は、Word、Excel、PowerPoint といったオフィス製品群へ生成 AI を統合する製品です。2024年後半には Fortune 500 の約70%が採用したと Microsoft は述べています。s4-51 s4-52
sec-3 で見た「要約や下書き」から一歩進んで、メール、会議、表計算、スライドという日常業務の真ん中に AI が入ってきた形です。もちろんこれは「ホワイトカラーがすぐ代替される」を意味しません。多くは仕事の一部を少し速くする形ですが、「文章生成ツール」から「日々の業務ソフトの一部」になった意味は大きい。

もっとも直接的に労働に触れたのは、やはりコーディングツール市場でした。
GitHub Copilot は 2024年初頭に 130万人超の有料加入者を抱えていると Microsoft が明かしています。s4-53 Claude Code、Cursor、Codex 系ツール、Gemini CLI が並走し、「AI にコードを書かせる」はもう珍しいことではなくなりました。

ここで大事なのは、AI が仕事を完全に奪ったかどうかではありません。
仕事の工程のどこまでを任せられるか が、じわじわと広がっていること。文章の叩き台、コード生成、バグ修正、テスト、ブラウザ操作。ひとつひとつは限定的でも、つながると、補助と代行の境目はかなり曖昧になります。

2024年秋から2026年春までのボトルネックをまとめるなら、自律性と、その自律性をどう測るか でした。

最先端モデルはすでに、一部の分野では人間の補助を超え、まとまった成果物を返し、複数段階の仕事をこなし、研究開発にも部分的に入り込んでいます。けれど最初から最後まで信頼して丸投げできるかと言えば、まだかなり怪しい。だからこそ評価が難しく、同時に、その伸びがいっそう不気味に見えるのです。

sec-1 から sec-4 までを振り返ると、流れは一本の線でつながっています。

  • まず、事前学習が汎用モデルの土台を作った
  • 次に、スケーリングと API が市場を作った
  • その後、ChatGPT が社会の入口を壊した
  • そして今、推論・エージェント・事後学習によって、能力向上のサイクル自体が速くなり、AI が研究開発にも関与し始めている

ここまで来ると、自然にひとつの疑問が浮かぶはずです。
この延長線は、どこで止まるのか。
