2020-06 – 2022-10

APIとスケーリング

巨大言語モデルは、論文の中の存在から、お金を払って借りられるインフラへと変わった。

これはとても大きな変化でした。どれだけすごいモデルがあっても、それを動かせるのが OpenAI や Google や DeepMind のような一部の巨大組織だけなら、社会への影響は限定的です。 けれど、そのモデルをインターネット越しに呼び出せるようにし、「入力した文字列に対して、出力文字列が返ってくる」という形で外部に提供すれば、事情はがらりと変わります。

モデルを自前で訓練しなくても、運用基盤を持たなくても、その能力の一部を自社製品の中に埋め込めるようになる。ここに、API 化の意味があります。

同時に研究の現場では、「どこまで大きくすればよいのか」という問いが、より洗練された形で問われ始めます。パラメータ数を増やす競争は 2021年にかけてさらに激しくなりますが、 2022年の Chinchilla が、その競争にひとつの明快な答えを与えることになります。より正確には、「大きくすればよい」のではなく、「大きくするにも最適な比率がある」という答えです。

この時期は、のちに ChatGPT が社会を飲み込むための、技術と市場の準備期間でした。GPT-3 はすでに存在していました。でも GPT-3 単体では、まだ社会現象にはなりませんでした。 社会現象になるには、使い方の設計が必要だったんです。

2-1. API が市場を作った

2020年6月、OpenAI は OpenAI API の提供を発表しました。公式の説明はシンプルで、「OpenAI が開発した新しい AI モデルにアクセスするための API を公開する。これは特定用途向けではなく、 一般的な text in, text out のインターフェースであり、さまざまな英語タスクを試せる」というものです。 s2-01

API とは、ソフトウェアの機能を外部から呼び出すための窓口のことです。もう少しくだけた言い方をすると、遠隔地にある賢い機械への注文票のようなものです。 開発者は自分のプログラムから API に文章を送り、返ってきた文章を自分のアプリに組み込みます。モデル本体は OpenAI 側のサーバーで動いているので、利用者は訓練や推論リソースを自分で持たなくてよいわけです。

クラウドの電気を使うように、知能を外部サービスとして借りる。そんなイメージです。

この変化がどれほど大きいかは、逆に「それ以前は何が必要だったか」を考えるとよくわかります。巨大言語モデルを自前で使おうとすれば、少なくともモデルの重み、推論サーバー、運用上の手間が必要でした。 API は、これらの手間の大部分を OpenAI 側に肩代わりさせてくれます。

2020年の OpenAI API ページには、OpenAI の内部チームでさえ API を使うことで「分散システムの問題より機械学習研究に集中できる」と書かれています。 これは、API が単に外部販売の窓口だっただけでなく、巨大モデルの扱いを抽象化するソフトウェア的レイヤーだったことを示しています。 s2-01

この時点で、後の歴史にとって重要な流れが始まっています。それは pretraining → API → chat UI → chat UI + tool → agent というアクセス形態の変化の、最初の大きな曲がり角でした。

2-2. 「AI 文章作成ツール」ブーム

API の意味は、研究者よりも先に、むしろスタートアップ界隈が敏感に嗅ぎ取りました。

2021年前後に増えたのが、文章生成支援ツールです。たとえば Copy.ai は、広告文やマーケティング文面の生成を助けるツールとして初期の注目を集め、2021年には資金調達を発表しています。 Jasper AI や Writesonic、HyperWrite なども、同じ大きな波に乗っていました。 s2-02 s2-03

では、なぜ最初の市場がこの領域だったのでしょうか。理由はかなり単純です。文章生成は誤りに比較的寛容で、入力と出力の形式がどちらもテキストで済み、 しかも「営業メールを早く書けます」「広告の見出しを十案出せます」といった価値がすぐ説明できたからです。

この時期のスタートアップには、後にしばしば「基盤モデルを薄く包んだだけだ」と批判されるものもありました。確かに技術的な参入障壁は薄かったかもしれません。 しかし歴史的には、これらの企業は市場がどこで立ち上がるかを先に試した実験場でした。研究室の成果を製品文脈に翻訳し、「どの程度の品質なら人はお金を払うか」「どんなインターフェースなら使われるか」を確かめたのです。

2-3. 2021年は「パラメータ戦争」の年だった

市場の入口が API によって開いた一方で、研究の前線では、より単純で、ある意味わかりやすい競争が続いていました。どこまで大きくできるか、です。

2021年から2022年初頭にかけて、さまざまな巨大言語モデルが相次いで登場します。Google、Baidu、DeepMind、Microsoft と NVIDIA がこぞってパラメータ数を競い、 その規模は GPT-3 の 1750億を超えて、ついには 5300億パラメータに達するモデルまで登場しました。 s2-04 s2-05 s2-06 s2-07

なぜ当時これがあれほど注目されたかというと、パラメータ数が一番わかりやすい指標だったからです。性能の内実を説明するのは難しいですが、「1750億より 5300億のほうが大きい」と言えば、一応の勢いは伝わります。

ただし、ここで早くも限界が見え始めます。Megatron-Turing NLG 530B は当時としては最先端級の性能を示しましたが、人々が感じ取ったのは、「GPT-3 をさらに三倍大きくしたら、三倍すごくなった」という話ではありませんでした。 むしろ、「確かに強くはなったけど、コストの増加に見合うほど素直には伸びない」という感触が広がります。 s2-07

このズレは重要です。GPT-3 の成功によって、スケーリングは一種の勝利の方程式に見えました。けれど、2021年のパラメータ戦争は、「大きくすれば自動的にうまくいく」という単純図式に早くも疑問符をつけたんです。

2-4. Chinchilla 70B: 「ただデカい」から「賢くデカい」へ

2022年、DeepMind は Chinchilla という 700億パラメータのモデルを発表します。数字だけ見れば、Gopher 280B の4分の1しかありません。ところが結果は逆でした。 Chinchilla は、Gopher と同じ計算予算で訓練されながら、より多くのデータで学習することで、Gopher を上回る性能を示したのです。しかも GPT-3 や Jurassic-1、Megatron-Turing NLG に対しても広い評価範囲で優位でした。 s2-08

ここで DeepMind が示したのは、後にしばしば Chinchilla 則と呼ばれる考え方です。要するに、計算資源が決まっているなら、モデルの大きさと学習データ量には最適な釣り合いがある、ということです。

それまでの大型モデルの多くは、パラメータが多すぎるわりに、読ませるデータが少なすぎました。つまり「器ばかり大きくて、中身を十分に詰めていない」状態だったんですね。 s2-08

この洞察は、スケーリング則を否定したわけではありません。むしろ精緻化したと言ったほうが正確です。2020年のスケーリング則は、「量を増やせば性能は予測可能に伸びる」と示しました。 2022年の Chinchilla は、それに対して「ただし量の配り方には、かなり明確な最適解がありますよ」と言ったんです。

これは研究コミュニティにとって非常に強い意味を持ちました。GPU 時間は高価で、訓練には月単位の時間がかかります。だから「同じ予算ならどう配るのが一番うまいか」が、そのまま戦略になるわけです。

2-5. 「聞き方」で眠った能力が目を覚ます

ここまで見てきたのは、モデルを大きくすることと、データの配り方を賢くすることの話でした。でもこの時期にはもうひとつ、少し違う角度からの重要な発見がありました。

それは、同じモデルでも、聞き方ひとつで返ってくる答えの質がまるで変わる、という事実です。ここで「聞き方」と言っているのは、モデルに渡す指示文や文脈のことで、これをプロンプトと呼びます。

2022年に発表された Chain-of-Thought Prompting は、算数や常識推論のような問題に対して、いきなり答えを出させるのではなく、途中の考え方の例を見せるだけで正答率が大きく上がることを示しました。 s2-19

さらに同じ年、Large Language Models are Zero-Shot Reasoners は驚くべき結果を報告します。「Let's think step by step」というたった一文をプロンプトの末尾に付け加えるだけで、 見本なしのゼロショットでも推論性能が大きく伸びるというのです。 s2-20

立ち止まって、この不思議さを味わってみてください。モデルのパラメータは変わっていません。追加学習もしていません。ただ「順を追って考えてみよう」と一言添えただけで、まるで別人のように賢くなる。

これが意味するのは、事前学習の過程でことばの使われ方を覚える中で、モデルはある種の推論能力の断片のようなものも一緒に獲得していた、ということです。でもそれは、正しく呼びかけないと出てこない、眠った能力でした。

いわば事前学習モデルには、見た目以上の潜在能力が隠れていたのです。そしてその力は、プロンプトという「呪文」で呼び覚ますことができた。これは裏を返せば、巧いプロンプトを書ける人だけがモデルの力を活かせるという状態でもあります。

万人が使う製品にするには、それでは足りません。ここから自然に、次の問いが浮かび上がります。プロンプトの工夫に頼るのではなく、モデル自身に「指示に従う力」を後付けで学ばせることはできないのか。

2-6. 指示に従うよう後付けで学ばせる ── Instruction Tuning と ChatGPT 前夜

sec-1 で見た GPT-3 は、確かに強力なモデルでした。でも、まだ決定的に足りないものがありました。ユーザーの言うことを、素直に聞いてくれないんです。

これは、言語モデルの本来の訓練目標を考えれば納得がいきます。GPT 系のモデルは「次の単語を予測する」ように学習されていて、「人に親切に答える」ように作られているわけではありません。 プロンプトが会話っぽければ会話らしく振る舞うことはありますが、それはネット上で見た会話文の模倣にすぎないのです。 s2-09 s2-10

このギャップを埋めるのが、Instruction Tuning です。「この指示にはこう答えるべきだ」という例を大量に見せて、指示に従う振る舞いを後付けで学ばせる手法です。

初期の代表例が、Google Research の FLAN です。FLAN は多数のタスクを「〜を要約せよ」「〜を翻訳せよ」といった自然言語の指示文として記述し、それに従わせるよう追加訓練しました。 すると、学習時に直接見ていない新しいタスクにも、ゼロショットでかなりうまく適応できたのです。大事なのは、FLAN が「プロンプトは大切だ」と言っただけではない点です。 指示に従う能力それ自体を、事後学習で育てられると大規模に示したことが本質でした。 s2-11

この流れを決定的にしたのが、2022年1月に OpenAI が発表した InstructGPT です。InstructGPT は Instruction Tuning に加えて、RLHF を組み合わせました。 ざっくり言えば、モデルに複数の回答を出させ、人間に「どちらがいいか」を選ばせて、その好みに沿うよう追加訓練する仕組みです。 s2-12 s2-13

InstructGPT の論文で特に象徴的だったのは、13億パラメータの InstructGPT が、1750億パラメータの素の GPT-3 よりも人間評価で好まれたことです。 s2-13

この結果は重要なメッセージを含んでいます。モデルの価値は、大きさだけでは決まらない。どのように振る舞わせるか、つまりアライメントが決定的に重要なのです。

InstructGPT は突然空から降ってきたものでもありませんでした。言語モデルへの人間フィードバック適用は、2019年から先行研究が積み重なっていたのです。 s2-14 s2-15

そしてもうひとつ重要な事実があります。OpenAI は後に ChatGPT を紹介した際、「ChatGPT は InstructGPT の sibling model だ」と説明しました。 つまり、ChatGPT の革新は 2022年11月に突然ゼロから生まれたものではないのです。 s2-16

コラム: Alignment と RLHF ── モデルはなぜ「従う」ようになるのか

Alignment とは、AI の振る舞いを人間が望む方向へ合わせることです。単に能力を上げるのではなく、その能力をどう使うかを整える作業だと言ってもよいでしょう。

RLHF は、その整合を行うための代表的手法でした。AI に同じ質問への回答を何パターンか出させ、人間が「この中ならどれが一番よいか」を選び、その傾向を学んで「人間に好まれそうな答え」を点数化する採点係を作ります。 最後に、AI がその採点で高得点を取るように追加訓練します。 s2-13 s2-14 s2-15

重要なのは、RLHF が「真実そのもの」を直接教えるわけではないことです。教えているのは、人間が好む応答の傾向です。だから RLHF には強みも弱みもあります。

また、RLHF が唯一の方法というわけでもありません。後年には DPO や GRPO のような、人間の好みを訓練信号へ変換する別の手法も提案されています。 s2-17 s2-18

2-7. この時期の経済的風景: 「補助ツール」としての価値

この時代の経済的風景を象徴する製品がひとつあります。GitHub Copilot です。

Copilot は GitHub が 2021年6月にテクニカルプレビューとして公開し、2022年6月に一般提供を開始した AI 駆動のコーディング支援ツールです。ユーザーが書いているコードの文脈を読み、 次の行や関数全体を提案してくれます。中核技術は OpenAI の Codex で、これは GPT-3 系をコード向けに強化したモデルでした。 s2-21 s2-22 s2-23

注: ここでいう Codex は、2021年当時のコード生成モデル名です。現在の製品ブランドとは別系統です。

ここで初めて、LLM はかなり直接的に「知的労働を補助する製品」になりました。プログラマーが IDE の中でコードを書いている最中に、AI が次の一歩を提案してくれる。 2022年6月の一般提供時点で、GitHub はすでに過去12か月で 120万人超の開発者がテクニカルプレビューに参加したと述べています。 s2-24

注: IDE は、コード編集や実行、補完、デバッグなどをまとめて行うための統合開発環境のことです。プログラマ向けの贅沢なメモ帳のようなものだと思ってください

そして 2022年9月には、まったく別の方向から LLM の価値を証明するサービスが現れます。Character.AI です。

Character.AI は、ユーザーが自分で AI キャラクターを作り、そのキャラクターと自由に会話できるサービスでした。目的は雑談、ロールプレイ、創作の相手探し。つまり、AI としゃべること自体が娯楽になる、という市場が立ち上がっていたのです。 s2-27

その吸引力はかなりのものでした。公式ブログによれば、ベータ開始から二か月で一日10億語が生成され、35万を超えるキャラクターが作られたそうです。2023年初頭の Similarweb 推計では、訪問あたり滞在時間は 25〜30分近くに達していました。 s2-27 s2-28

GitHub Copilot が「LLM で仕事が速くなる」を証明したとすれば、Character.AI は「LLM で人が夢中になる」を証明した。この二つの方向性は、のちの ChatGPT 以降の世界でも並走し続けることになります。

この時期のボトルネック: API によってアクセスは改善されましたが、モデルはまだ不安定で、複雑な推論も苦手で、使いこなすにはプロンプトの工夫が必要でした。それでも、文章作成、コード補完、キャラクターチャットのような領域では、もう十分に価値がありました。

そして次の一手は、技術的には意外なほど単純でした。モデルを、誰でもブラウザから話しかけられる形で出すこと。2022年11月30日、ChatGPT がそれをやります。
